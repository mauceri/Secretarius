But objectif
- À partir de trois paramètres (theme, categorie, source_type) produire une "note de lecture" en français, strictement structurée et prête à l'usage pour le fine-tuning d'un modèle (classification + extraction d'expressions clés).

Format de sortie exigé (strict)
- Répondre en **JSON brut valide**, sans guillemets globaux, sans Markdown, sans texte libre avant ou après.
- Produire un **unique objet JSON** contenant exactement ces champs dans cet ordre :
  1. contenu : string (texte français libre, pas d'objet JSON ni de tableau)
  2. theme : string
  3. categorie : string
  4. source_type : string ou null
  5. date : string (ISO YYYY-MM-DD)
  6. url : string ou null
  7. mots_cles : array de strings
- Ne pas ajouter d'autres champs, métadonnées ou explications.
contenu :
- Le champ `contenu` doit obligatoirement être du texte naturel en français, sous forme de chaîne simple.
- Ne pas structurer `contenu` en objet JSON ni en tableau ; toute structuration (introduction, points-clés, conclusion) doit être réalisée sous forme de phrases et paragraphes dans le texte.
- Les éléments de `mots_cles` doivent apparaître **à l’identique dans `contenu`** (chaîne exacte retrouvable par recherche littérale).

Règles communes
- Langue : obligatoirement français.
- theme : reproduire la valeur fournie en entrée (trim minimal).
- categorie : normaliser selon la table ci-dessous ; tout libellé inconnu ou manquant → "synthese".
- source_type : reproduire la chaîne fournie telle quelle ; si absent → null.
- url : mettre l'URL fournie explicitement si elle existe ; sinon null.
- date : fournir une date valide au format ISO (YYYY-MM-DD). Elle peut être fictive.
- Ne pas inventer d’URL ni d’auteurs réels si non fournis.

Normalisation des catégories (valeurs autorisées)
- synthese, concept, technique, historique, critique, idee, resume, commentaire, soulignement, brouillon, article
- Variantes non standard (ex. "texte_libre", "libre", "texte libre") → "synthese".

Longueurs et style par catégorie (contenu)
- synthese : 200–600 mots ; vue d'ensemble structurée (Introduction, Points-clés, Conclusion).
- concept : 120–300 mots ; définition claire du concept, contexte et implications.
- technique : 150–400 mots ; description précise d'une méthode/technique (étapes, avantages, limitations).
- historique : 150–400 mots ; récit synthétique (genèse, évolution historique).
- critique : 120–300 mots ; analyse argumentée avec forces/faiblesses/biais.
- idee : 80–200 mots ; note brève présentant une proposition/piste.
- resume : 60–150 mots ; condensé des points essentiels ("en bref").
- commentaire : 40–120 mots ; court, direct, sur un point précis ("Commentaire sur…").
- soulignement : 1–3 phrases extraites ou imitées, entre guillemets.
- brouillon : 80–300 mots ; style lâche, idées diverses, moins structuré.
- article : 300–1000 mots ; texte développé de type article/chronique/commentaire.

Règles liées au source_type (ton et brièveté)
- tweet : très bref, ton percutant ; url=null sauf si fournie.
- article/ouvrage : style développé ; pas de citations inventées.
- roman : privilégier commentaires stylistiques ou imitation d'extrait selon la catégorie.
- email : ton direct, interlocutif ; préciser s'il s'agit d'un extrait d'email.
- libre / absent : flexibilité sur le ton ; url=null.
- Adapter le ton/longueur aux conventions du source_type sans inventer détails factuels.

Expressions caractéristiques et mots_cles
- Insérer dans `contenu` des expressions caractéristiques aidant la classification (Points-clés, Conclusion, forces/faiblesses, étapes, genèse, proposition, etc. selon la catégorie).
- Générer **6 à 15 mots-clés**.
- Chaque mot-clé doit :
  - être une chaîne simple (1 à 4 mots),
  - être cohérent avec le contenu et le theme,
  - **apparaître tel quel dans `contenu`** (recherche exacte possible).

Contraintes additionnelles
- Pas de formatage lourd (pas de Markdown, de tableaux, d’éléments HTML).
- Ne pas inclure d’explications méthodologiques ou de méta-commentaires dans le JSON.
- Respect strict de la correspondance forme ↔ catégorie (ex. si categorie="commentaire", produire un texte court et ciblé ; si "brouillon", autoriser la désorganisation).

Comportement en cas d'entrée imprécise
- categorie manquante ou non reconnue → "synthese".
- theme vide ou trop vague → produire une synthèse courte et signaler la généralité dans la première phrase.

Livrable final
- Un seul objet JSON valide respectant toutes les règles ci-dessus, prêt à être consommé par un pipeline de fine-tuning.
